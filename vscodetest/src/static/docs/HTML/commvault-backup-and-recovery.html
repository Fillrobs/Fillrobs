CommVault Backup and Recovery in CloudBolt

Learn how to set up CommVault in CloudBolt, import CommVault VM groups, and begin backing up your VMs.

Introduction

CommVault manages plan association for VMs via VM groups, using filters to add new Servers to predefined groups.

The CloudBolt CommVault integration allows you to import these CommVault groups, so that the choice on the Blueprint is one of a group to assign the vm. After the group is chosen and the Order is submitted, CloudBolt adds the newly created Server to the VM group by hostname.

Once you’ve created a Server with a CommVault backup plan configured, you’ll need to run a new backup job on the VM Group in CommVault. Use the same credentials you used to add the backup plan to log into the console.

Prerequisites

  • An existing Resource Handler in CloudBolt for one of CommVault's supported Cloud Providers.
  • Configure CommVault
    • Storage
    • Server Backup Plan
    • VM Groups
    • Hypervisor
    • Plan configured in CommVault and assigned to a VM Group.
We recommend that your Resource Handler is configured to use a fully-qualified domain name (FQDN) on your server as opposed to using an IP Address.

Configure CommVault Data Protection in CloudBolt

Create a CommVault Integration in CloudBolt

  1. In CloudBolt, click Admin > Integrations > Data Protection.
  2. Click +Add a Data Protection Solution...
  3. Choose CommVault. 
  4. CloudBolt will reach out to CommVault and create a connection based on the information entered in the following fields:
    • Name: A descriptive name for this data protection solution.
    • IP address*: The IP Address or FQDN for your CommVault instance. Do not enter the protocol prefix (e.g. https).
    • Account username*: Username for the CommVault account.
    • Password*: Password for the CommVault account.
  5. Click Create.

Import VM Groups

Once your CommVault connection is created in CloudBolt, import the VM Groups.

The Protection Plans tab displays the VM Groups from CommVault. Admins must manually update this list when changes are made in the CommVault interface.

  1. Click Import Protection Plans, which will import VM Groups from CommVault.
  2. Select from the list of available VM Groups to import. This list is generated based on the groups that an admin has already configured in the data protection solution.
  3. Once a plan is imported, view it in the Protection Plans tab. Click X to remove a plan from the list.

Add the Data Protection Plan Parameter to Environments

Next, set up Data Protection for Environments by adding the Data Protection Plan parameter to them. You can also pre-define the Parameter selection in the Blueprint.

Provision a Server with a Protection Plan

You can predefine the CommVault Data Protection group for a Server at order time, or let the user choose on ordering. CloudBolt will add the Server to the selected CommVault VM group.

Predefine a Data Protection Plan

  1. In CloudBolt, create a Blueprint or Edit an existing Blueprint.
    1. If editing an existing Blueprint, click Build tab > Configured Environments > Edit the Environment.
  2. In the "Blueprint Item Parameters for [Your Environment]" form, click the Data Protection Plan toggle to Predefine and select a plan from the dropdown.
  3. Click Save.
  4. On the Overview tab, click Order.
  5. Kick off the initial back up in CommVault, then backups will be taken according to the schedule set in CommVault's Server Backup Plan.
Parameters that have a single value provided by the Environment will not be shown to users at order time; the value will simply be included once the Blueprint is ordered.

Restoring from Snapshots

Visit Restoring from Snapshots for information on restoring CommVault snapshots in CloudBolt in the scenario of a data loss event.

When you click Restore on a Server in CloudBolt, a job gets submitted and CommVault which starts the restore process.

This job can take time depending on the amount of data/connectivity etc. In CloudBolt, a pop up message says “Host has been restored”; wait for the restore process to complete on CommVault before checking the system is restored.


Concept: How Servers are Restored in CommVault and CMP

There are various restore types in CommVault, many of which center around restoring individual files. In CMP, the Restore functionality that has been implemented is a VM Restore, in place with unconditional overwrite. This means that the whole virtual machine is replaced with a new virtual machine of the same configuration based on the specified backup.

The key take away is that a restore is really a replacement. It does not internally modify the contents of the VM, but instead replaces it. In the example where we restore “dns1.mydomain.org”, that machine for inventory purposes is now a new machine with the same name. After the next CommVault backup it will have the same id: 23, but a new GUID.

During the next VM sync, CMP will find the new machine and retire the old one. We do not add Data Protection on VM sync, so the newly imported virtual machine will not have the Data Protection tab and will not have access to its backups, although they are still in CommVault.

If you delete “dns1.mydomain.org” in CMP and re-provision a machine of the same name, you will find that CommVault has tracked the life cycle of this host name and you will get the same client id and have access to all the old backups of this host name.

Concept: Understanding the Data Protection Lifecycle

During provisioning, CloudBolt adds the hostname of the Server to the Content of the specific CommVault VM Group. This virtual machine will not exist in CommVault until the next backup is run on the VM Group.

Backup requests do not queue. Requesting a backup on a Group or client while a backup is running will result in an error.


Managing Virtual Machines

When a virtual server is discovered during a backup, it is assigned a Client ID and a GUID. At this point it can be individually managed.

CommVault manages the life cycle of a virtual machine based on its host name. If you provision a machine named “dns1.mydomain.org”, that machine will get an integer Client ID; for example 23, and the Client ID will persist across restores. The GUID will change with each instance of the machine.

Outcomes

CommVault integration in CloudBolt is complete and your provisioned Servers are being protected.
