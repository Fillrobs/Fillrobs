Blueprints from Remote Source

Learn about Blueprints from Remote Source and how to set them up.

Introduction

CloudBolt offers the ability to have Blueprints populated from a definition stored in a remote source location, such as GitHub, gitlab, or a filesystem. This feature may be useful for a variety of reasons, such as having multiple CloudBolt instances (e.g., dev, staging, and production) pointing to the same Blueprint definition or making use of source control features (e.g., history, reversion).

Preparing a Blueprint to Become Remote Source

There are two ways to create a Remote Source Blueprint. These instructions assume you are working with a source Blueprint created in CloudBolt.

  1. Configure a source Blueprint in CloudBolt with all of the desired attributes and components and then creating a flat-file version of it to store in your remote source.
  2. Develop the entire flat-file description of a Blueprint in your remote source, then configure a Blueprint in CloudBolt to point to that remote URL.

Create a Flat-File Blueprint

After setting up the source Blueprint as desired in CloudBolt, start the process of creating a flat-file version of it to store in the remote source by exporting it. Be sure to do so with instance-specific information to capture all the details.

The export will create a zip file, which will need to be fully-extracted before being put in the remote source location. Start by unzipping the main zip file. If there are actions used in the BP they will have their own sub-zip files, which will also need to be unzipped. Any Resource Actions will have two layers of zip files to unzip. You will wind up with a folder structure like the example below, which you can then put in a folder in your remote source location.

Example
remote_source_folder
|--bp_name.json
|--list_image.png
|--base_action_name
  |--base_action_name.json
|--resource_action_name
  |--resource_action_name.json
  |--base_action_name
  |--base_action_name.json
We recommend using a new export, even if you already have an export for the Blueprint you will use as a source. In particular, pre-8.7 and pre-8.8 exports will be missing a critical piece of information used for action Blueprint items and management Resource Actions, respectively: the name of the base action associated with them.

Configure a Blueprint to Use Remote Source

Once you have a flat-file Blueprint definition stored in a Remote Source location, configure a Blueprint in CloudBolt to point to that Remote Source URL and use it to populate the Blueprint. You can either configure an existing Blueprint or create a basic new Blueprint and set it to use Remote Source immediately after creation.

  1. Navigate to your Blueprint in CloudBolt.
  2. On the Overview tab, click Configure Remote Source.
  3. Enter the Remote Source URL. The URL should be the path to the raw version of the Blueprint JSON file (bp_name.json in the folder structure above).
    • Only CloudBolt Admins have permission to configure it, since doing so can create CloudBolt plug-ins and otherwise be very impactful.
  4. Enter a Remote Source Password. See Secrets Stored in Blueprints.
  5. When the Remote Source URL is set, the Blueprint is automatically populated with the data from the definition stored in the Remote Source.
  6. Use the refresh button that appears next to the URL to update the BP from then on. An infotip displays the last time the Blueprint was refreshed from that Remote Source URL.

Once the Remote Source URL is set, the Blueprint becomes read-only in CloudBolt, where the only changes that can be made are to modify its Remote Source URL or delete it. Removing the Remote Source URL causes it to return to being a regular Blueprint.

Connection Info for Remote Source

Connection Info objects used for Remote Source (Blueprints, Actions, etc.), must be named "Source Code Repository" in the Connection Info form.

GitHub for Remote Source

Using GitHub for the Remote Source requires one to create a connection to GitHub - pointing to the JSON file as described above will not work with GitHub.

  1. Log in to GitHub and create a Personal Access Token in the GitHub repository.
    • The Access Token can be created with NO scopes.
    • Save the generated key. This key will be needed below.
  2. Create a ConnectionInfo, refer to the example form below:
    Field Name Value / Notes
    Name Source Code Repository
    IP/Hostnamegithub.com
    Port443
    Protocolhttps
    UsernameType in your GitHub Username.
    PasswordType in your GitHub Password.
    User Auth HeadersCheck this checkbox.
    Headers Replace <your personal access token here> below with the Personal Access Token generated from the Step 1.
    Copy Code
    JSON
    {
      "Authorization": "Bearer <your personal access token here>"
    }
  3. Click Save
  4. Now you will be able to refer to Blueprints under Source Control in GitHub.

Secrets Stored in a Remote Source Blueprint

A prompt will show to optionally enter a password to encrypt any Blueprint secrets when “Configure Remote Source” is clicked. For more details, see Secrets Stored in Blueprints.

Upgrading to 9.2 or greater

After upgrading to CloudBolt version 9.2 or greater, Remote Source Blueprints that contain secrets must be reconfigured. If the following steps are not completed, any secrets in your Remote Source Blueprint will become blank on its next refresh post-upgrade.

  1. Re-export Blueprints from the original source CloudBolt and provide a password for encryption.
  2. Update the definition stored in your remote source location.
  3. Update every Remote Source Blueprint that is populated from that definition and enter the same password in the “Configure Remote Source” dialog for decryption.

Updating a Remote Source Blueprint

There are a couple ways of keeping a Blueprint up-to-date with its definition in Remote Source, namely manual refresh and a Recurring Job called ‘Refresh Blueprints from Remote Source’ that can do it on a schedule. The safest option is to use a combination of the two, but CloudBolt does not mandate a particular approach.

Manual Refresh

If relying on manual refresh, whenever a change is made to the definition of the Blueprint stored in the Remote Source, someone must tell CloudBolt to pull the latest definition of the Blueprint from the Remote Source URL. This can be done either by a Blueprint manager viewing the Blueprint in the web UI and pressing the refresh button or a user with API Access POSTing to the refresh endpoint on that Blueprint (append a ‘/refresh’ to the API URL for the Blueprint).

Some advantages of this approach are that you can ensure the refresh happens immediately after a change in the Remote Source and it will be very apparent if something goes wrong with the refresh. The main disadvantage is the fact that it’s manual and could be forgotten. Also, it must be done by a user with the proper permissions, either to manage the BP or access the API, which could be a bottleneck. These downsides can be mitigated by making use of the Recurring Job.

Refresh Blueprints from Remote Source Recurring Job

CloudBolt comes out-of-the-box with a disabled Recurring Job called Refresh Blueprints from Remote Source, which will refresh all Blueprints configured with a Remote Source URL when run.

To make use of this automated, broader approach to refresh, enable the Recurring Job and make sure it has a schedule that is appropriate for your use case.

The advantages of this option are that it is automatic and will generally do a pretty good job of keeping Remote Source Blueprints up-to-date if scheduled appropriately, for all users. Disadvantages include not being guaranteed that the BP will match the remote source definition when used in CB, if it was changed since the last run of the job, and requiring an extra step to confirm the job is succeeding, to avoid missed failures. These can be mitigated by also employing manual refresh. One additional caution: ensure that updates to the remote source definition, at the URL configured, are only allowed if it’s OK for them to be pulled into CloudBolt (e.g., if the URL points to a master branch in GitHub, review changes on a branch and approve before merging them to master).

Limitations

In this beta version of this feature, there are some limitations to be aware of, which fall into two different categories.

Should Not be Included in Remote Source Blueprint

The following will currently cause issues with refreshing from the Remote Source if they are included in the definition of the Blueprint stored there:

  • Container Object build items
  • Load Balancer build items
  • Sub-Blueprint build items

Please only use this feature with Blueprints that do not include the above elements.

Will Not be Populated from Remote Source

The following will not cause any problems if they’re included in the definition of the Blueprint stored in Remote Source, but will not be used on refresh and therefore will not appear on the local version of that Blueprint:

  • Build or Teardown Items of type External Flow
To protect the integrity of existing actions that may also be used for other things, refreshing a Blueprint from Remote Source will create any missing actions used in build items, teardown items, discovery plug-ins, and the Management tab, but not modify existing ones. Options for how to avoid impacting other uses of the actions while still allowing the remote source definition of a Blueprint to modify them are being considered. We suggest configuring any actions to use remote source code when originally preparing a Blueprint to become remote source.
