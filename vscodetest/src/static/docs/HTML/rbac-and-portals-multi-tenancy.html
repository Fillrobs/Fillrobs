Overview

Multi-tenancy in CloudBolt enables one CloudBolt server to be used by multiple different organizations, while ensuring the privacy of their data and resources. This allows CloudBolt Administrators to set up a set of global policies, automations, Blueprints, Actions, and report globally on usage across all customers. Multi-tenancy provides a tailored hybrid cloud experience to each Tenant organization.

  • Multi-tenancy is especially useful in the case of large companies with different lines of business that should not see each other's Servers, Blueprints, and Environments.
  • It is also useful in the case of Managed Service Providers (MSPs) who provide bespoke cloud services to their external customers. These organizations need peace of mind of knowing that their customers (internal or external) can only access resources they're permitted to, and cannot interfere in any way with each others' operations.

Multi-tenancy features

Role Based Access Control (RBAC)

CloudBolt's Role-Based Access Control features a Group tree which allows administrators to model different internal and external customers. Each Group can be assigned a set of Group Administrators who can further subdivide their organization into sub-Groups.

SSO and LDAP Integration

While the native RBAC capabilities of CloudBolt provide the framework for Multi-tenancy, it is best practice to leverage an existing organization’s Group structure and authentication mechanisms with a single user directory source.

CloudBolt allows for several forms of integration with external authentication and directory services. Generally speaking it is preferred that an organization leverages delegated authentication (where CloudBolt does not store passwords, but instead queries an identity provider to see if a user should be logged in) and secondly, synchronization of a user’s group from a directory source.

Combined, these two integrations enable external credentials management and allow the admins of those identity providers to manage CloudBolt permissions based on Group memberships. This is important for large, Multi-tenant deployments requiring centralized management of authentication and authorization.

Today, CloudBolt supports any LDAP source for both authentication and group mapping as well as a number of SSO endpoints. For example, as of this writing CloudBolt has been successfully integrated with Active Directory, Okta, Azure LDAP/SSO, Ping One, ADFS, GSuites, and even home grown SAML-based Identity Providers (or IDPs).

It should also be noted that CloudBolt provides administrators even more flexibility by exposing trigger points in the external user sync and SSO user update processes allowing you to add custom business logic to derive a user's entitlements.


Per-Group customized Order Form

Each Group in CloudBolt can decide which questions Users should be asked when ordering Servers, what the options and constraints should be for those, what attributes should be set on Servers without asking the User, and what networks should be available.

In the above example, the hostname template and power schedule are predefined to a set value for this Group, while the expiration date can be set to anything up to 30 days from the order time, the Server size option is constrained to small and medium, and the quantity is required to between 1 and 5. This data-driven approach to generating the order form is crucial for providing a tailored user experience.

Note that there are additional parameters that may be assigned to groups that do not appear on the order form. For example, consider a VM instance provisioned by CloudBolt in AWS, which must be tagged with a specific Cost Center code. This value can be stored as a parameter on the Group and applied using the bi-directional tagging capabilities.

Branded Portals

CloudBolt allows for customization of the look & feel of the user interface based on the URL used to get to that CloudBolt instance. For example, a managed service provider who uses one CloudBolt as the portal for multiple external customers could give each customer a different URL to use the same CloudBolt, and the user interface would appear tailored to each customer.

Each portal can have a different set of authentication methods associated with it so that tenants only see their own domains as options on the login page.

Per-group catalog items (Blueprints)

Anyone in a group in CloudBolt who has the blueprint.manage permission can create Blueprints and share them with others in their Group. This empowers each organization to create their own catalog of services and automate the deployments and tasks that are most important to them.

Access to Blueprints can be clearly defined under the Groups tab of each Blueprint.

Note the distinction between being able to Deploy (order) a Blueprint and/or Manage (edit).

Conclusion

These Multi-tenancy features within CloudBolt enable Tenants to tailor their own experience and gain agility, while freeing administrators to focus on setting policies for their hybrid cloud infrastructure and other high business value projects.
