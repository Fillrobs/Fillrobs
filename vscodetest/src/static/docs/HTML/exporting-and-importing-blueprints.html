Exporting and Uploading Blueprints

Blueprints can be exported as files in order to be easily reused across CloudBolt instances or shared in the CloudBolt Forge, a repository of sample CloudBolt content. Exported Blueprints are typically stored as zip files containing JSON-formatted metadata about the Blueprint along with related items and actions.

Exporting Blueprints

  1. In CloudBolt, click Catalog. 
  2. Select the Blueprint you wish to export, or use the API to export a Blueprint.
  3. On the Overview tab, click Export...
  4. A prompt to enter a password to encrypt any secrets will show when a Blueprint is selected for exporting. See Secrets Stored in Blueprints below.
  5. Choose between a sanitized or instance-specific version of the Blueprint, where the sanitized version will omit potentially sensitive information or details that may not make sense when importing into a different CloudBolt, such as Environments, Groups, and Global IDs.
  6. The Blueprint will export as a zip file to be uploaded in another CloudBolt instance or Content Library.

Exported Blueprint Contents

Exports contain all build and teardown Blueprint items, any discovery configuration, all Blueprint-specific resource actions, and all action scripts used by these. Exports of Blueprints that create a Resource will contain all the necessary information on the selected Resource Type. 

Exports will also contain all the necessary details on any Blueprint-level parameters and, if exported with instance-specific info, enabled environments for Blueprint items along with any predefined parameters.

Exports of Blueprints that create a Resource will contain all the necessary information on the selected Resource Type. The upload process will look for an existing matching Resource Type to use, and create a new one if none is found.

Check out the CloudBolt Forge for some example Blueprints.

Uploading Blueprints

Blueprints can be uploaded from the Catalog page or imported using the API, and have an option to either replace the existing Blueprint or create a new one.

  1. In CloudBolt, click Catalog.
  2. Click Upload.
  3. Click Choose File.
  4. If the Blueprint was exported with a password to encrypt secrets (see step 4 of Exporting Blueprints above), enter it here to decrypt secrets on upload. 
  5. Check Replace Existing if desired. See tip below.
If you’re relying on the Global ID feature for referencing Blueprints through the API and want to maintain consistency across CloudBolt instances, you’ll generally want to select Replace existing when uploading Blueprints. If Replace existing is not selected and a Blueprint already exists with the same name as the export, two Blueprints will exist with different Global IDs on one CloudBolt.

Uploaded Blueprint Contents

Actions

Unless you choose to replace the existing Blueprint, new versions of all Actions will be created on upload. If you’re uploading with the Replace Existing option, CloudBolt will look for Actions that match the attributes appropriate to that specific type of Action and use those if they exist. Action Items will not change on upload if they are updated/changed on the exported Blueprint.

Because only CloudBolt Admins can create or edit CloudBolt Plug-ins, if a Blueprint manager who is not an Admin tries to upload a Blueprint that contains one or more CloudBolt Plug-ins, they will receive a permissions error and not be able to do so.

Similarly, because non-CB Admins cannot create or edit shared Actions, they will not be able to upload a Blueprint that contains a shared Action or would replace an existing shared Action (because they’ve chosen to replace existing), regardless of the Action type. Such users can still import Blueprints that do not contain any CloudBolt Plug-ins and would not create or replace shared actions, and CB Admins can import all Blueprints as long as they also have Blueprint management permissions.

Parameters

The import process for Parameters will look for existing matching Parameters to use, and create new ones if none are found.

Environments

Importing of enabled Environments will only happen if an Environment with the same name is found. This is due to the more complex, instance-specific, and less user-driven nature of Environments.

Groups

Groups will behave the same way as environments.

 Secrets Stored in Blueprints

When a Blueprint is selected for export or configured to use a remote source URL, any secrets (Parameter values with the type ‘password’) stored in that Blueprint can be encrypted with a user-designated password. When the Blueprint is imported, another prompt will appear to enter the user-designated password. This decrypts secrets when the Blueprint is imported.

This is useful for maintaining security when sharing exported objects across different CloudBolt instances, or when configuring a remote source Blueprint.

If a user-designated password is not entered in the prompt, a default key will be used. In this case, the exported Blueprint’s secrets can only be decrypted when uploaded to the same CloudBolt instance it was exported from.

When exporting/importing through the API, secrets are encrypted/decrypted with the default key only. We recommend using the CloudBolt UI to provide a user-designated password.
