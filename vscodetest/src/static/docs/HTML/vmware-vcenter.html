VMware vCenter

VMware vCenter is a server management solution that provides a centralized platform to control your VMware vSphere environments. This platform allows you to automate and deliver virtual infrastructure across the hybrid cloud. CloudBolt integrates with vCenter versions 6.5 → 7.x

To Set up the vCenter Resource Handler:

  1. Go to Admin > Resource Handlers.
  2. ​Click +Add a New Resource Handler...
  3. Select VMware vCenter.


Account Username & Password

The credentials for the vCenter account that CloudBolt will use to perform server creation, power management, synchronization, deletion, and other actions.

Clusters

Clusters are required to deploy using CloudBolt. If you are attempting to deploy to a single node, first create a cluster using vCenter. A CloudBolt Environment will automatically be created for each cluster selected, and the cluster-specific options (such as the resource pools and datastores) will be imported.

Templates

  1. From the vCenter Resource Handler, click Templates.
  2. Click Import Templates. The templates in this list are populated from the vCenter server CloudBolt is currently connected to.
  3. Check the box next to the template(s) you wish to import, then click Import selected templates.
Imported templates show in the Templates tab, along with their OS Build, OS Build Environments, Disk Size, Credentials, VMware Tools Status, and Content Library status.

Networks

CloudBolt communicates with VMware using the standard HTTPS connection on port 443. Network rules must allow traffic on port 443 between the CloudBolt server and all vCenters and ESX hosts.

Network Adapter Type

The Network Adapter Type of a vCenter template in CloudBolt overrides the Network Adapter Type of newly provisioned VMs in vCenter when a vCenter server is provisioned in CloudBolt. This may also override the Adapter Type on the template in vCenter.
  1. To change a Network Adapter Type, navigate to Admin > Integrations > Networks.
  2. Click the Network Name > Edit.

VM Customization

By default, CloudBolt will use the compatible guest customization tools in vCenter templates to customize newly provisioned VMs with parameters such as hostname, IP address, DNS server, WINS servers, domain membership, and more.

Windows Sysprep answer files, often called unattend.xml, can be passed to vCenter from CloudBolt to provide finer-grained control over how a Windows VM is customized. To use this feature, add the parameter labeled “Sysprep Answer File” to a vCenter-backed Environment, and set its contents to the XML stored in a Sysprep answer file. This file is templatized by CloudBolt, so dynamic values such as {{ server.hostname }} can be used.

Using a sysprep answer file will ignore the default guest customizations parameters described above, therefore if using an answer file, ALL customizations for a VM must be specified as part of the supplied answer file.

Tags

vCenter tags are used to filter VMs into Resource Handlers. For VMware, CloudBolt will treat all tags as if they have single cardinality, meaning there can be only one tag value per category per VM. If you make a category in VMware that supports multiple tags, only one of those tag’s values will show on the server in CloudBolt. Also, it’s important to be aware that if you were to change the value of the parameter associated with that tag in CB, all the other values would be removed from that VM.  

vCenter Tags do not sync bidirectionally. If you completely remove a tag from vCenter, it will not be removed from CloudBolt, even upon a Resource Handler sync. Only tags that are added in vCenter are synced to CloudBolt, not removed tags.

Disks

Add Parameters to the vCenter Environment to customize Disk choices. Options include

  • Minimum Free Space
  • Provisioning Type
  • Datastore

VMware Datastore Free Space Parameter

Add a single option to the Environment. This is most relevant when you're adding an additional disk to an existing Server (from its Disks tab). If there’s not a value for the “VMware Datastore Free Space” Parameter on the Server (e.g., because it was discovered, or provisioned before the Parameter was configured on the Env), then CloudBolt checks the Environment and will use the first option found on it, which is why there should only be one option. Otherwise, the behavior might be surprising (nothing shows the Parameter when adding the disk).

If there’s no value for the Parameter on the Server and no option for it on the Environment, CloudBolt will check for a Global Default and use that value as the constraint. Neither Global Options nor options on a Group will be used when adding a Disk.

Snapshots

vCenter Servers support snapshots. See Snapshots for set up and use details.
