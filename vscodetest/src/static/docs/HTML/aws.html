When an AWS Resource Handler is created, CloudBolt provides a choice of Regions to be imported, and one CloudBolt Environment will be created for each. Additionally, VPCs can be specified for each CloudBolt AWS Environment to build and manage instances within a specific VPC.

Enhanced multipayer account capabilities are also available through the use of the AWS Identity and Access Management (IAM) Role integration. See set up step 5 in Setting up the AWS Resource Handler.

Setting up the AWS Resource Handler

  1. Go to Admin > Resource Handlers.
  2. Click +Add a new resource handler...
  3. Select Amazon Web Services.
  1. Enter your Access Key ID and Secret Access Key. See the Security Credentials section of the AWS Console.
  2. Optionally, enter the ARN for the Identity and Access Management (IAM) Role you wish CloudBolt to assume.
    1. This is defined in the target AWS, on the My Security Credentials page under Roles. See AWS Identity and Access Management on Amazon's docs for greater detail.

AWS Servers Detail View

AWS Servers will display the IAM Profile ARN in the Amazon Web Services Details section of the Server Details, when an ARN exists for that server.

Environments

Each AWS Environment comes with tech-specific out-of-the-box Parameters and options to allow for easeful deployment and customization of AWS instances from CloudBolt. Networks and AMIs may also be imported for each Environment.

Find Tech-Specific AWS Parameters on the “AWS Parameters” tab on the Environment:

AWS Windows Instances

AWS for Windows has a few differences when running remote scripts.


AWS Windows VMs

Windows VMs on AWS allow providing a password for running remote scripts. This can be provided as described in the Remote Scripts Passwords section.

Either a password or SSH key may be given, depending on the VM and desired behavior. If a password is provided, CloudBolt will simply use it. If not, CloudBolt will attempt to obtain the Administrator password from EC2 using the provided SSH key, but that operation will only succeed if the VM is configured correctly in EC2 for get_password_data to be successful.

By default, AWS relies on SSH Keys to access VMs.

AWS Tech-Specific Script Execution

To run Remote Scripts on AWS without entering a username/password, enable Tech-Specific Script Execution.

Images

When you import AMIs in CloudBolt, CloudBolt will create an OS Build for that AMI. If you import the corresponding AMI in other regions, CloudBolt will automatically reuse the pre-existing OS Build, and associate the newly imported AMI with it. This will only happen if the name of the OS Build has not been altered and matches the name of the AMI. However, CB Admins can edit the OS Build for an AMI to override that default behavior.

Importing and Viewing AWS Billing Data

In CloudBolt, you can import your AWS billing data and view billed costs for individual servers and CloudBolt groups.

Set Up

  1. First, set up an S3 bucket in AWS to have billing reports exported as .csv files. See Setting Up an Amazon S3 Bucket for AWS Cost and Usage Reports.
  2. In CloudBolt, go to the detail view of the AWS Resource Handler associated with the account for which billing is being exported.
  3. Click  Edit Settings > enter the Name of the S3 bucket under S3 billing bucket.

The recurring job Import Public Cloud Billing Data is set to automatically run once a month. You can configure its cron schedule to run at an opportune time after bills come in, or just run the job manually to sync existing billing data.

Viewing

View billing summaries on the Billing tab of AWS Servers, AWS Resource Handlers, and Groups.

  • Groups Billing tabs will display a graph with data aggregated per-Resource Handler, where you’ll see the charges for EC2 instances within each Resource Handler for that group.
  • Servers will show billed charges for the instance only, excluding charges for any associated resources.
  • CloudBolt will also recommend reserved instances that can be purchased through the EC2 portal which can save money for individual instances. View these recommendations from the Billing tab on your Resource Handlers.

Provision to EC2 Dedicated Hosts

CloudBolt provides the ability to provision AWS VMs directly to one of your EC2 Dedicated Hosts. If you would like the option to provision AWS VMs to an EC2 Dedicated Host, follow the steps below.

Note: This feature is enabled using Tech-Specific Parameters for AWS. Choose a value for either Dedicated Host or Dedicated Host Group. If both are given values, Dedicated Host will be the default.

Prerequisites

If using Dedicated Host in CloudBolt:

  • An AWS Dedicated Host.
If using Dedicated Host Group in CloudBolt:
  • An AWS Dedicated Host Group with correct licensing.
  • At least one licensed AMI that is allocated to a Dedicated Host. This is required to provision into a Host Group.
    • Make note of the licensed AMI ID, as you will need it in CloudBolt.​
    • In AWS, search for License Manager > Customer Managed Licenses.

Set up Dedicated Hosts in CloudBolt

  1. In your AWS Resource Handler, click Import AMIs and select the AMI that is not licensed.
    1. If you select an AMI that is associated with a license configuration, during provisioning AWS will assume you want a Host Group and will give an error.
  2. Select an Environment.
  3. Navigate to the AWS Parameters tab.
  4. Click Import on the Parameter to bring in values from AWS. Once a Host is added to a Host Group in AWS, it will not appear in the Host Parameter options.
  5. Choose a value for Dedicated Host.
  6. On the Order Form, select an option for Dedicated Host.

Set up Dedicated Host Groups in CloudBolt

  1. In your AWS Resource Handler, click Import AMIs and select the AMI that is licensed.
  2. Select an Environment.
  3. Navigate to the AWS Parameters tab
  4. Click Import on the Parameters to bring in values from AWS.
  5. Choose a value for Dedicated Host Group.
  6. On the Order Form, select a value for Dedicated Host Group.
If there are no options for the Dedicated Host or Dedicated Host Group Parameters in a particular Environment, either because they haven't been imported yet or none exist, the Order Form will display a freeform text field. An error will show if an invalid object is entered.

Modifying or Adding Volumes

AWS has some specific Parameters associated with volumes that other Resource Handlers do not have. In addition to Disk Size, you may also need to specify Parameters around EBS Volumes.

  • If you do not specify an EBS Volume Type it will default to gp2.
  • If adding an io1 volume, you will also need to specify an amount of IOPS which must be greater than 100.
  • When creating a gp2 volume, the size must be greater than 4GB. When creating sc1 or st1 volumes, the size must be greater than 500GB.
  • You can also specify if you want the Volume to be encrypted with AWS standard encryption.

When placing a modification order to add or change disks in the API, these are the accepted Parameters:

disk_size: The size to change the original volume to
new_disk_size: The size for the new volume to be added
ebs_volume_type: The ebs volume type for the new volume, currently accepted values are [io1, gp2, gp3, st1, sc1, standard]
iops: Accepted when using io1 volume type
aws_volume_encryption: Submitted with True or False Boolean Value to encrypt the additional volume
