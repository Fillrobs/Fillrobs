Setting up Multi-Tenancy in CloudBolt

Now that you have learned about the concepts and use cases of Multi-Tenancy, learn how to set it up in CloudBolt.

  • Perform Basic Setup
  • Test the Multi-Tenancy Setup
  • Security Recommendations
  • Optional Setup Steps
  • Outcomes

Prerequisites

  • CloudBolt version 9.4.5 or newer.
  • Licensed for Multi-Tenancy
  • A CB Admin User and two Users you can log in as who are not CB Admins.

Perform Basic Setup

  1. Obtain a Multi-Tenancy license file from your account representative at CloudBolt.
  2. Log in to the CloudBolt instance UI as a CB Admin.
  3. Click Admin > Maintenance > CloudBolt Product License. 
  4. Under Upload a New License, paste the Multi-Tenancy License.
  5. Click Apply License.
  6. Click Admin > System > Multi-Tenancy.
  7. Create two Tenants, adding at least one User to each of the new Tenants as a Tenant Admin.
  8. Navigate to the Groups page. 
    1. Create two top-level Groups to represent the two Tenants. The Groups can have the same name as the Tenants.
    2. Add the two Tenant Admin Users to the corresponding Groups. If you do not know which Roles to add them to on the Group, add them to all Roles on the Group.

Test the Multi-Tenancy Setup

Tip: To test changes to Multi-Tenancy without logging in and out of CloudBolt, we recommend using either three separate browser windows, or a browser extension that allows multiple sessions (like Multi Session Box for Chrome or Multi-Account Containers for Firefox). 
  1. Log in to three different browser sessions:
    1. One as a Global CloudBolt Admin.
    2. One as Tenant Admin from the first Tenant.
    3. One as Tenant Admin from the second Tenant.
  2. As a Tenant Admin for the first Tenant, navigate to Admin > Resource Handlers.
    1. Create a new Resource Handler. This Resource Handler will automatically be assigned to the first Tenant as it was created by a User who belongs to that Tenant.
  3. As a Tenant Admin for the second Tenant, navigate to Admin > Resource Handlers and confirm the Resource Handler created above is not visible.

You can repeat these steps for different kinds of objects: Container Orchestrators, Blueprints, etc.

Important: Blueprints are governed by Group permissions rather than Tenant membership, but since you have separate top-level Groups for the Tenants, they will not be seen across Tenants (with the exception of Blueprints that a CB Admin makes available to all Groups or multiple Groups across different Tenants).

Security Recommendations

  1. If you upgraded CloudBolt from a version earlier than 9.4.5, we recommend you remove the ability for Resource Admins to make their Blueprints available to all Groups. 
    • Click Admin > Roles, edit the Resource Admin Role and remove blueprint.any_group_can_deploy permission. See Group Permissions for Blueprints for more information.
  2. We recommend disabling the Cost Dashboard Report to remove the potential of leaking information across Tenants.
    • Click Admin > Miscellaneous Settings. Scroll to the "Disabled out-of-the-box reports" field and add Cost Dashboard.
  3. To avoid Servers being built with the same default password in multiple Tenants, remove the default password for new Windows VMs.
    • Click Admin > Parameters > New Password and remove the global default. This will need to be set at another level (Group or Environment) for Windows VM provisioning in VMware Resource Handlers to work.

Optional Setup Steps

Set up Branded Portals

If you want each of your Tenants to have a differently-branded or white-labeled experience, create a Branded Portal for each Tenant.

  1. In CloudBolt, click Admin > Portals and Branding.
  2. Create a Portal for each of your Tenants.
  3. Navigate to your Tenant's detail page.
  4. Click Edit > Select the Portal you created for it.

When a Tenant User logs in to the Portal designated above, that Portal's branding will be used for the UI. If there is no Tenant associated with the User, or no Portal associated with the Tenant, CloudBolt will select a Portal using its normal logic based on the host part of the URL. CloudBolt will use the Default Portal if there is no match.

Add Tenant Roles

You may grant different Users different levels of access within the same Tenant. For example, grant Users permissions to only manage Configuration Managers and Environments.

  1. In CloudBolt, click Admin > Security > Multi-tenancy.
  2. Add a new Tenant Role, select the appropriate permissions, and assign Users to it.
Note: The assignment of Users to Tenant Roles can also be done when adding Users to a Tenant in the UI.

Modify Tenant Roles

For some organizations, there may be a use case to grant Tenant Admins all permissions except a few; for example, the ability to create and manage Users. 

In this case, we recommend modifying the Tenant Admin Role that CloudBolt comes with and removing the permissions you do not want to grant. Upon upgrade, CloudBolt will not add new permissions to this role. If CloudBolt introduces new Tenant permissions in future versions, those new permissions will need to be added manually to this or other Tenant Roles after upgrading.

Automatically Associate Users with a Tenant

For Users within a Tenant who do not need administrative privileges, it is not strictly necessary that they have a Tenant set on them. Their Group membership will keep them from seeing anything that belongs to Groups of which they are not members.

However, there are at least two reasons why you may want the Users to belong to a Tenant:

  1. If the User manages Blueprints and needs to use Tenant-specific OS Builds in their Blueprints.
  2. If you want the Tenant-specific Portal to be used after they log in, but you do not have separate hostnames set up for each Tenant that are pointed at the CloudBolt instance.

Add Users to a Tenant

Users can be added to Tenants from the Tenant details page, with no Tenant Role if you do not want them to perform Administrative actions.

To automatically assign all Users to a Tenant, update the External Users Sync or SSO User Update Plug-in:

  1. In CloudBolt, click Admin > Actions > Orchestration Actions > Other.
  2. Navigate to the SSO User Update or External Users Sync Plug-in and modify it to suit your needs.

Outcomes

Begin using Multi-tenancy in CloudBolt to enable certain Users to administer aspects of CloudBolt within the confines of their particular organization.
