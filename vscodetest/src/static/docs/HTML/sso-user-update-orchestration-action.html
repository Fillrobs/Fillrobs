SSO Orchestration Actions

Learn how the SSO User Update Orchestration Action works when run using SSO/SAML.


If there is an enabled SSO User Update Orchestration Action, it will run on the appropriate User immediately after that User is created or identified. The SAML Assertion coming from the SSO IdP will be validated.

User Orchestration Actions Workflow

  1. CloudBolt runs the incoming User through any enabled SSO User Update Orchestration Actions.
  2. CloudBolt runs the same User through any enabled External User Sync Orchestration Actions as described in External User Sync.
  3. The out-of-the-box SSO User Update Action will update certain fields on the User, based on the information from the SSO IdP and fields defined as Required Attributes on the SSO Provider.

Note: The out-of-the-box Action code will only update fields not included in the Required Attributes for this SSO Provider.

  • For example, if an SSO Provider defines “email” and “username” as Required Attributes, then only an existing User’s first_name and last_name will be updated to match the information received from the SSO IdP when this action runs.

Configuring SSO Users with Actions

Configure CloudBolt to handle everything related to SSO Users:

  • Entirely in an SSO User Update action.
  • Entirely in an External Sync User action.
  • In a combination of the two.

The default SSO User Update Orchestration Action provides simple handling for associating a User with a particular LDAP domain when they are created/updated during SSO login. To do anything other than the default behavior, set the desired behavior in either the SSO User Update or External User Sync Orchestration Actions.

Username and LDAP

The out-of-the-box Orchestration Action has two different methods for handling automatically assigning a User to a particular LDAP within CloudBolt:

  1. The SSO IdP sends the username attribute:
    • Format: <username>@<ldapdomain>.
    • The Action will set the incoming User's username to <username> and associate the User with the LDAPUtility that has an ldap_domain that matches <ldapdomain>.
Example

If an SSO IdP is configured to send the username "herbert@cloudbolt.org", this user would end up with the username "herbert" and be associated with the LDAPUtility that has the domain "cloudbolt.org". This example assumes such an LDAPUtility exists.


  1. Explicitly define the domain name of the LDAP Utility for a given User in the ldapDomain attribute field on your SSO IdP.
    1. Change the name of the field that CloudBolt expects from the SSO IdP by changing the global variable LDAP_DOMAIN_SSO_ATTR_NAME in the SSO User Update Orchestration Action to something other than ldapDomain.
Example

If the SSO IdP sends the LDAP domain for a user in a field called UserLDAPIdentifier, then update the line of the SSO User Update Orchestration Action where LDAP_DOMAIN_SSO_ATTR_NAME is defined to LDAP_DOMAIN_SSO_ATTR_NAME = "UserLDAPIdentifier".

Then, if the SSO IdP sent a SAML assertion to CloudBolt with the data: username: "herbert" and UserLDAPIdentifier: cloudbolt.org, CloudBolt would create or update a User to have the username "herbert" and associate that User with the LDAPUtility that has the domain "cloudbolt.org".


External User Sync / LDAP

External User Sync Orchestration Actions will run as part of CloudBolt’s Assertion Consumer Service whenever CloudBolt logs an SSO user in. See Using the “User Permission Sync From LDAP” Action.

The External User Sync actions, if any are enabled, will run after the SSO User Update Actions (see SSO User Update Orchestration Action above). If the User is being logged in via an SSO Provider, then the External User Sync action will receive the attrs_from_sso kwarg (the same one that the SSO User Update action receives), which is a Python dict containing all the key:value pairs sent from the SSO IdP to CloudBolt.


Orchestration Action Arguments

These arguments are sent to SSO User Update Orchestration Action during SSO login:

Argument Description
job The Job associated with this Action. Will only be populated if the Action is being run as part of a Job. This will be None when called as part of an SSO login.
user The User being updated or created.
attrs_from_sso A dict of all the key:value pairs sent from the SSO IdP.
sso_instance The database model for the SSO Provider on CloudBolt. 
creating A boolean flag that indicates if the User above is being created (True) or updated (False).
