Customizing Terraform Orchestration Actions

Learn how to customize the way Terraform is called and how resources are integrated with the CloudBolt Platform.


The manner by which Terraform commands are invoked are exposed as Orchestration Actions, allowing Administrators to customize both how Terraform is called and how resources are integrated with CloudBolt.

There are two distinct, conceptual pathways in which Terraform is invoked:

  • The “constructive” pathway, where resources are provisioned using Terraform (“Provision”).
  • The “destructive” pathway, whereby resources that were provisioned by Terraform are destroyed (“Destroy”).

Access Terraform Orchestration Actions

  1. In CloudBolt, click Admin > Actions.
  2. Search Terraform to show the Terraform Actions.
  3. Click any Action Name to view the associated code and details.
The functions defined in these Actions are called directly by the CloudBolt Orchestration Engine. The function name, function signature, and output types must remain consistent, or CloudBolt's execution of Terraform will fail.  


Considerations

Like other Orchestration Actions in CloudBolt, those specific to Terraform can be scoped to different contexts, such as limiting an Action to a subset of Groups.
CloudBolt's execution of Terraform will fail, most likely with a traceback, if there is a Disabled Action at any given Trigger Point.

Provisioning Resources with Terraform

When creating resources in CloudBolt via a "Terraform Plan" Action (e.g. via a Blueprint), CloudBolt executes the following steps:


Terraform Pre-Provision

This step creates required state that is consumed by downstream steps.

  • The enabled action must define a pre_provision() function.

Terraform Init

This step is responsible for sending flags and environment variables to terraform init. If the “Init” step fails, the “Provision Failure” step will be executed to handle the failure.

  • The enabled action must define an init() function.

Terraform Plan

This step is responsible for sending flags and environment variables to terraform plan. If the “Plan” step fails, the “Provision Failure” step will be executed to handle the failure.

  • The enabled action must define a plan() function.

Terraform Apply

This step is responsible for sending flags and environment variables to terraform apply. If the “Apply” step fails, the “Provision Failure” step will be executed to handle the failure.

  • The enabled action must define an apply() function.

Terraform Post-Provision

This step performs teardown and integration logic after executing terraform apply. For example, the list of virtual machine resource types, defined by Terraform Providers, can be configured to associate resources provisioned by Terraform as “Server” objects in CloudBolt.

  • The enabled action must define a post_provision() function.

Terraform Provision Failure

If any of the “Init”, “Plan”, or “Apply” steps fail, this step is run to aide in cleanup and teardown. Out of the box, this Action will delete the CloudBolt “Resource” object that’s associated with the “Order”.

  • The enabled action must define a provision_failure() function.

Terraform Provision Cleanup

This is the final step in the “Provision” workflow, and runs regardless of success or failure of any upstream action. Out of the box, this step deletes the Terraform plan file that’s written to disk.

  • The enabled action must define a provision_cleanup() function.


Deprovisioning Resources with Terraform

When deleting resources that were created in CloudBolt via a “Terraform Plan” Action (e.g. via a Blueprint), CloudBolt executes the following steps:


Terraform Pre-Destroy

This step creates required state that is consumed by downstream steps.

  • The enabled action must define a pre_destroy() function.

Terraform Destroy

This step is responsible for sending flags and environment variables to terraform destroy. If the “Destroy” step fails, the “Destroy Failure” step will be executed to handle the failure.

  • The enabled action must define a destroy() function.
Note: terraform init and terraform plan -destroy are run before the “Destroy” step. These behaviors are not exposed to users. Any changes to the “Terraform Init” or “Terraform Plan” steps in the “Provision” workflow have no effect on the “Destroy” workflow. 

Terraform Post-Destroy

This step performs teardown and verification logic after executing terraform destroy.

  • The enabled action must define a post_destroy() function.

Terraform Destroy Failure

If the “Destroy” step fails, this step is run to aide in cleanup and teardown.

  • The enabled action must define a destroy_failure() function.

Terraform Destroy Cleanup

This is the final step in the “Destroy” workflow, and runs regardless of success or failure of any upstream action. Out of the box, this step deletes the Terraform state file that’s written to disk and the associated object in CloudBolt.

  • The enabled action must define a destroy_cleanup() function.
