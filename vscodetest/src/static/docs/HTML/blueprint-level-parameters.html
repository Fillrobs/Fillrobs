Associate Parameters with Blueprints

Learn how Blueprint-level Parameters affect the Order Form when ordering a Blueprint and how they affect the deployed Resources.

Introduction

Blueprint managers have the ability to associate Parameters with Blueprints using the Parameters tab when managing the Blueprint.

Parameters include both Custom Fields and Preconfigurations. These parameters will appear when ordering the Blueprint, and the values entered will be associated with each build item in the deployed resource, the deployed resource itself, or both, depending on the destination that the manager selects.

For example, this could be used to easily set the same Expiration Date for all servers provisioned by the Blueprint or to mark the entire resource as being associated with a particular ticket number.

Setting Blueprint-level Parameters

  1. In CloudBolt, click Catalog > "Your Blueprint".
  2. On the Parameters tab, set options and constraints for a parameter at the Blueprint level, just like you can do for Environments and Groups.
    The Parameter tab will not be visible if the Resource Type is None.

These options and constraints, however, only apply to the particular Blueprint and take precedence over any options or constraints set somewhere else.

Considerations

  • Blueprint-level parameters with Build Item destinations will always override overlapping parameters on the item. If a Blueprint-level parameter includes build items as a destination, it will not appear on the form when configuring or ordering individual build items in that Blueprint. This applies to both Custom Fields and Preconfigurations.
  • If a Preconfiguration is set at the Blueprint-level, then both the Preconfiguration itself and the Custom Fields associated with it will also not show on any of the build item forms when being configured or ordered.
  • If a Custom Field is set on the Blueprint-level and a Preconfiguration with that Custom Field is also set on a Server Tier, the Blueprint-level value will override the Preconfiguration value for that one parameter.
    • For example, if 3 is entered for a Blueprint-level Mem Size but small is selected for a Server Size Preconfiguration (which translates to 1 CPU and 1 GB memory), the resulting server(s) will have 1 CPU and 3 GB memory.
  • To use a blueprint parameter in multiple build steps follow this process:

"Create Blueprint Parameters" Permission

The Create Blueprint Parameters (blueprint.create_parameters) permission allows non-Admin CloudBolt users to create parameters only for Blueprints they can access. The Manage Blueprints (blueprint.manage) permission is required to make use of the Create Blueprint Parameters permission. This permission is assigned to specific roles.

First, see if any of the out-of-the-box roles will work for this permission. See Groups and Permissions. Create a new role if needed, and assign users who will have the blueprint.create_parameters permission to this role.

Assigning to an Existing Role:

  1. Click Admin > Roles.
  2. Find the Role you wish to edit.
  3. In the Actions column, click the pencil icon to edit and open the Modify Role window.
  4. In the Permissions field, click the caret to select blueprint.create_parameters.
  5. Click Save changes.

Creating a New Role:

  1. Click Admin > Security.
  2. Click Roles.
  3. Click + New Role.
  4. Fill in the fields.
  5. In the Permissions field, click the caret to select blueprint.create_parameters.
  6. Click Create.

This new role is now available to assign to a Group and any of the users in that Group.
