Permissions and Objects of Multi-Tenancy

Learn about Tenant Permissions, Group Permissions, and the objects in CloudBolt that can be used with the Multi-tenancy feature.

  • Introduction
  • Tenant Permissions
  • Global Roles
  • Tenantized Objects
  • Outcomes

Introduction

Before the introduction of Multi-tenancy in CloudBolt, Group Permissions governed which Users could see and manage Servers and Resources, deploy to specific Environments, etc. The Group Permissions still play that role when Multi-tenancy is enabled.

Multi-tenancy introduces finer granularity for the permissions that come with the CB Admin Global Role. In this way, Users can be granted access to manage a subset of the Admin objects only within the context of their Tenant.

For a complete setup of Multi-tenancy, both Group Permissions and Tenant Permissions should be utilized.

Tenant Permissions

Since a Tenant Admin will only have permissions to see and manage certain Groups based on their Group Role membership, the Tenant Admin will only be able to provide the Environments they create to those specific Groups, and will not be able to see or modify Groups from other Tenants.

Available Tenant Permissions

This list is the complete set of available Tenant Permissions:

  • Manage Resource Handlers
  • Manage Configuration Managers
  • Manage OS Builds
  • Manage Applications
  • Manage Networks
  • Manage IPAMs
  • Manage Preconfigurations
  • Manage SSH Keys
  • Manage Environments
  • Manage Container Orchestrators
  • Manage Users

Global Roles

In addition to Group and Tenant permissions, CloudBolt has several Global Roles.

The three Global Roles (CB Admin, Global Viewer, and DevOps admin) apply across Tenants and should not be granted to Tenant Admin users, but reserved for users who are expected to perform work across all Tenants in CloudBolt.

Tenantized Objects

CloudBolt supports associating certain objects with Tenants, and allowing Tenant Administrators to list, view, create, modify, and delete these objects, which will be referred to as "Tenantized objects".

This list is the complete set of objects that are available to associate with a Tenant:

  • Resource Handlers
  • Configuration Managers
  • OS Builds
  • Applications
  • Networks
  • IPAMs
  • Preconfigurations
  • SSH Keys
  • Environments
  • Container Orchestrators
  • Users
  • Resource Pools

Branded Portals

Portals can be assigned to Tenants from the Tenant Details page, but they are not considered Tenantized objects since Tenant Admins are not given access to see and manage them. Viewing and managing Branded Portals is a setup step handled by CB Admin users.


Images / Templates

Images and Templates are specific to Tenants, but only indirectly via their Resource Handler. For example: If an Image is imported for a Resource Handler in a Tenant, only users in that Tenant will ever see that Image.

Global Objects and Tenant-Specific Objects

Setting Tenant on an Object

When a User who belongs to a Tenant and has Tenant permissions creates any Tenantized object, that User's Tenant is automatically set as the Tenant on that object. This prevents the object from being seen by Users in any other Tenant.

CloudBolt Admins see and manage objects in any Tenant, and should take care not to relate objects from different Tenants with each other; for example: if a CB Admin adds an Environment belonging to "Tenant A" to a Group that represents "Tenant B", Users in "Tenant B" would be able to see and deploy resources into "Tenant A's" Environment.

Creating new Tenantized objects

When CB Admins create new Tenantized objects, they can choose a Tenant for that new object. This allows CB Admins to help Tenant Administrators in the setup process.

Once an object is created, its Tenant cannot be changed. Tenantized objects have many relationships to other objects, which has potential to cause data integrity issues. Instead, delete the Tenantized object and recreate it in its new Tenant.

Global objects with no Tenant

Global objects (those that do not belong to a Tenant) are usable by Tenants as they normally would be. The objects will not appear to Tenant Admins in the list view, and they will not be manageable by Tenant Admins.

Global OS Builds with Tenant-Specific Images

Global OS Builds behave differently than other Global objects. Because CB Admins typically want to make specific Blueprints available across Tenants, any of these Blueprints that contain Server Tiers need to use an OS Build that is global (not Tenant-specific). To deploy that Global OS Build to a Tenant-specific Environment, Tenants need to use an Image from a Tenant-specific Resource Handler. Therefore, Tenant Admins need a way to associate Tenant-specific Images with Global OS Builds.

To allow Tenant Admins to do this, Global OS Builds are included on the OS Build list page when viewed by a Tenant Admin (unlike other Admin list pages). They can also click on these Global OS Builds to get the details of the Global OS Builds, but they will not see the detail that a CB Admin would see. Other Tenants' Images that are associated with the OS Build will not be shown.

The only thing the Tenant Admins can change on a Global OS Build is to add their own Tenant-specific Image, which will enable them to deploy Blueprints that use that OS Build to their Tenant-specific Environments.

Tenant-Specific Unassigned Groups and Environments

When CloudBolt is running a "Sync VMs" Job and querying the results for new VMs, it puts those newly discovered Server records into a Group called "Unassigned" and an Environment called "Unassigned" (unless it can determine a single Environment that the Server belongs in, then places it in that Environment). A User with Permissions to the Unassigned Group is expected to reassign those Servers to the appropriate Group and Environment.

"Sync VMs" Job Discovered Servers in Multi-Tenancy

In Multi-Tenancy Enabled CloudBolt instances, it would not scale to have one central set of Users who reassigned the Servers for all the Tenants. When a new Tenant is created, CloudBolt automatically creates a new "Unassigned for <Tenant Name>" Group and Environment and places all discovered Servers that are within the Tenant's Resource Handler into that Group and Environment.

CloudBolt manages User permissions on these special Unassigned Groups, automatically granting reassignment permissions to any User who is assigned to that tenant (and revoking permissions if they are removed from the Tenant).

CloudBolt does not expose these Groups and Environments in the UI because there is nothing that needs to be done to these special Groups / Environments.

To see all VMs that belong to this Group or Environment:

  1. Click Servers > Show filters > set the Group and/or Environment filter > Apply.
  2. Select multiple Servers and reassign them with the [] Edit icon.

Tenant-specific Global SSH Keys

Tenant Admins can view, edit, and delete Global (not Resource Handler-specific) SSH Keys for their Tenant at Admin > SSH Keys.

Note: The Tenant cannot be changed on existing SSH Keys.

Add new Global SSH Key(s) from the list page, which will automatically be added to their Tenant.
