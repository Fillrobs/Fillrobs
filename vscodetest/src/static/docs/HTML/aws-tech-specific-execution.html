AWS Tech-Specific Script Execution

Learn how to use Tech-Specific Script Execution to run Remote Scripts on AWS without entering a username and password.

Introduction

AWS Tech-Specific Script Execution allows users to run Remote Scripts on AWS when CloudBolt cannot directly access or run commands on the virtual machine being targeted. This accomplished by setting AWS System Manager and running Remote Scripts using the Run Command feature of AWS. AWS Systems Manager is an AWS service used to view and control your infrastructure on AWS, including the ability to remotely execute commands on an EC2 instance.

AWS Prerequisites

Before you begin, observe the following requirements for your AWS configuration outside of CloudBolt.

  • AWS servers you choose for SSM must have the SSM (Systems Manager) agent installed. See AMIs with SSM Agent Pre-installed.

Linux

  • Linux EC2 instances must either have Port 443 exposed, or be using an appropriately configured VPC endpoint for SSM to work.

Windows

  • Windows EC2 instances require Port 3389 open for SSM to work. The usual launch-wizard default security groups will not work.
  • It is recommended to use an AWS Image Type of t2.small or greater for best performance results.
  • Windows systems take a few minutes longer than Linux systems to spin up and install SSM. The instance might say it is "Running" even before SSM is fully installed and initialized. The Instance System log in AWS can be helpful here.

Systems Manager VPC Endpoint Configuration

  • If your deployment does not have internet access through Port 443, configure your VPC by way of VPC Endpoints. See AWS Docs on Creating VPC Endpoints.

AWS Roles and Policies

Roles in AWS is the hinge upon which SSM in CloudBolt operates. CloudBolt needs permission to fetch / create the EnablesEC2AccessSystemsManagerRole from AWS. This role will be used to automatically enable SSM on certain operating systems. See Supported Operating Systems in AWS docs.

Note: Ensure that there is only one IAM Role per EC2 instance in AWS. 

Add the Required AWS Policy

The AmazonSSMManagedInstanceCore Policy is required to be added to the EnablesEC2AccessSystemsManagerRole in AWS.

  1. In the AWS Console, click IAM under Services.
  2. In Access management, click Policies.
  3. Use the Search to find AmazonSSMManagedInstanceCore and click the name to open the details.
  4. Click Policy usage. Make sure EnablesEC2ToAccessSystemsManagerRole is attached.
  5. If it is not attached, click Attach.

Setting up AWS Systems Manager in CloudBolt

Setup Overview

  1. Add the SSM Role to AWS.
  2. Add the Tech-Specific Script Execution Parameter to:
    • A Group / Environment.
    • A Blueprint.
  3. Provision a Server. 

Add the SSM Role to AWS

  1. In CloudBolt, click Admin > Resource Handlers.
  2. Select an existing AWS Resource Handler, or create a new one.
  3. On the Overview tab, click Create SSM Role.
This will either fetch an existing role from AWS (EnablesEC2ToAccessSystemsManagerRole), or create one with the appropriate permissions.
Note: The AmazonSSMManagedInstanceCore policy will be added, which is an AWS Managed policy containing a number of SSM and EC2 permission levels.

Add the Tech-Specific Parameter to Group / Environment

The Parameter can also be added to individual Servers. The following steps are still required in order to enable Remote Script SSM.
  1. Navigate to the Group or Environment that this AWS Resource Handler is associated with.
  2. On the Parameters tab, click + Add
  3. ​Find Use Tech-Specific Script Execution and add it to the Group or Environment.
  4. On the Parameters tab, click the Use Tech-Specific Script Execution parameter you just added.
  5. Ensure that Show on Objects and Available to all servers are enabled.



Note: If the Resource Handler is VMware, the Global setting is used.  

Set up a Blueprint to use Tech-Specific Execution

If you are using SSM as a part of the Blueprint, the EC2 will have that role associated with it. 

  1. In CloudBolt, click Catalog and open a Blueprint on which you will enable Tech Specific Execution.
  2. Click the Parameters tab. 
  3. Add the following Parameters: AWS IAM Role ARN and Use Tech-Specific Script Execution​.
  4. On the AWS IAM Role ARN Parameter, edit the Options column and add the ARN as an Option. The ARN can be found in Step 3 of Add the SSM Role to AWS in the imported/created Role string.



Provision a Server with Tech-Specific Script Execution

  1. Using the Blueprint created in Set up a Blueprint to use Tech-Specific Script Execution, provision a server.
The IAM Role must be correct for the given AWS Resource Handler, and use_tech_specific_execution must be True.
  1. On the provisioned server, verify that Tech-Specific Script Execution works by running an Ad-Hoc script.

Outcome

The Ad-Hoc script should successfully run without entering a username/password.

Amazon Machine Images (AMIs) with SSM Agent Pre-installed

The SSM Agent comes pre-installed on the following AMIs:

  • Windows Server 2008-2012 R2 AMIs (published in November 2016 or later)
  • Windows Server 2016 and 2019
  • Amazon Linux
  • Amazon Linux 2
  • Ubuntu Server 16.04
  • Ubuntu Server 18.04
  • Amazon ECS-Optimized

Troubleshooting

General

Remote scripts for SSM behave differently than typical remote scripts. A script on the remote server runs commands rather than the Remote Script running commands itself. When the server responds back, it could appear that the Remote Script was successful even if it failed.

  • Ensure that the script you are sending is properly configured, because errors may not return verbosely.
  • Make sure that exception handling is set up to check for errors.
  • Exception handling may be performed in the return code to check for errors.

Command Statuses

See AWS docs on Understanding Command Statuses.
