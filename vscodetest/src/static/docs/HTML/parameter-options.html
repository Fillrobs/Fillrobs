Parameter Options

Learn about Parameter Options in CloudBolt.

Introduction

Parameters may be limited to a static set of options on the Group, Environment, Blueprint for Blueprint-level Parameters, or globally. The options for a Parameter may also be generated by a Generated Parameter Options action (see Orchestration Actions).

How Option Order is Determined

In the context of Blueprint-level Parameters, those options take precedence. Next comes Group and Environment.

  • If both Blueprint and Group/Environment have options defined for a Parameter, CloudBolt will combine (take the union of) all of the options on the Group and Environment.
  • For a Parameter with generated options, those will be used unless there are also limited options configured on the Blueprint or Group/ Environment, in which case the overall options will be the intersection of the generated options and the other options.
  • When there are no options found by the process above, CloudBolt will use global options that are defined for the Parameter. If none of these are set, the user will be presented with a free-form entry field (confined within any constraints, as described below).

Parameter Override Precedence

The Parameter override precedence is determined in the following way:

Precendence
Global
Most general
Group -
Environment -
Order -
Server Most Specific


Sometimes, the process for determining options will decide that there is only one option available for a Parameter in that context. In that case, the Parameter will be considered to have its value provided and will not be shown on the order form.

Parameters that are not marked as being required will never have just a single option, due to the inclusion of a blank/ none option.

Global Parameter Defaults

Use global defaults to reduce configuration of individual Blueprint, Environment, and Group Parameters by setting a default value for a Parameter that can be automatically applied to orders.

Set a default value from a Parameter’s detail page. Configure a global default value to be automatically set on Servers, Resources, or both, as the ‘Global Target’. When setting a value for Servers, you can also constrain it to only apply to certain OS Families. Default values are overridden if the same Parameter is set on a Group, Environment, or Blueprint for a given order.

Parameter Constraints

Constraints consist of minimum, maximum, and/or regex match, depending on Parameter type. Global constraints are set in Admin > Parameters. Group and Environment constraints can be configured from the Parameters tab of the Group or Environment.

The order of precedence CloudBolt uses to determine which constraints apply goes from global (lowest precedence) to Group to Environment (highest precedence).

Example
As an example, imagine the Expiration Date is constrained to a maximum of 7 days globally, 5 days in the Acme Group, and 14 days for the San Jose QA Lab Environment.
When ordering a Server that is in neither in the Acme Group nor the San Jose QA Lab Environment, users will not be allowed to enter an Expiration Date more than 7 days in the future.

If ordering in the Acme Group and an Environment with no constraint for this Parameter, the Expiration Date must not be more than 5 days away. For a server in the San Jose QA Lab Environment, regardless of Group, the limit for an Expiration Date in the order dialog will be 14 days in the future. Note that these constraints apply to ordering Blueprints and setting Parameter values on servers in the same manner.

When a maximum is set on an integer or decimal Parameter, it will be rendered in forms as a slider, instead of a text field. If a minimum is not specified, a default minimum of 1 will be used for required Parameters, and a default minimum of 0 will be used for non-required Parameters. The amount that the slider increments can also be set for a given Parameter.

Parameter Dependencies

Dependencies may be set up between Parameters so that when a user enters a specific value in one input field, another will appear dynamically. Controlling Parameters can be of any type other than dates or code, and dependent Parameters can be of any type. One controlling Parameter can have multiple dependent Parameters. Ex. checking one checkbox could cause multiple additional fields to appear.

Example
  1. The Order Form could have an option for ‘Service Level’, with ‘Gold’, ‘Silver’, and ‘Bronze’ options.
  2. When ‘Gold’ is selected, additional options could be made to appear for ‘Enable Monitoring’, ‘Enable Backups’.
    1. If they check the ‘Enable Backups’ checkbox, another option could appear to ask how many days they would like to retain the backups.
To get started, navigate to Admin > Parameters and choose a Parameter that you want to make dependent on another. From there, you can create and manage a dependency.

Show/Hide Parameter Dependencies

Parameters can be configured to only show when a specific value is entered for another Parameter. You can set this up from the Dependencies section of a Parameter details page, by adding a controlling Parameter with a ‘show/hide’ dependency type. You can then specify whether you want the Parameter to show based on a specific value or have it show when any value is entered.

OS Family Parameter Dependencies

Parameters can also be configured to be dependent on OS Families, so they will only show up on the order form when an OS Build of a particular OS Family is chosen (ex. only show the ‘Windows Firewall’ Parameter when a Windows OS Build is chosen).

Regenerate Options Parameter Dependencies

A Parameter can be configured such that options that show up in one Parameter depend on the value selected for another Parameter.

For example, the options available for Microsoft Azure availability sets depend on which resource Group you are ordering with. Thus, these two Parameters come out of the box with a regenerated Parameter options dependency relationship, so that when you select a resource Group on the order form, the options for availability sets regenerate based on what is available for that resource Group. In this case, we would call ‘resource Group’ the ‘controlling’ field, and ‘availability set’ the ‘dependent’ field.


Set up a regenerated Parameter options dependency

  1. In CloudBolt, navigate to Admin > CloudBolt Objects > Parameters.
  2. Create two Parameters. One is the controlling field and the other a dependent field.
  3. Set up the dependency relationship:
    1. On the Parameter detail page for the dependent field, add a dependency to the controlling field.
    2. To do so, go to the Dependencies section in the bottom right of the detail page.
    3. Click the pencil next to the line stating This Parameter is not dependent on any other Parameters.
    4. A dialog will pop up with a form, where you may select the controlling field and ‘Regenerate Options’ for the dependency type.
  4. Set up a Generated Parameter Options Orchestration Action. After step 2, you will be prompted to either create a new plugin or select an existing one.
  5. Check that the base action is associated with the dependent field:
    1. When adding a new action, make sure that the dependent field is included under Custom Fields in the form.
    2. For existing actions, go to the base action, click Edit > add the dependent field to Custom Fields.
  6. Write your plugin code:
    • The plugin must define the get_options_list function and accept some keyword arguments, including field (dependent) and control_value (controlling) or control_values.
    • The options returned must be a list of tuples, and they can either be returned directly or in a dictionary under the ‘options’ key. See some examples below.

Manually Set up Generated Parameter Options

To manually set up a Generated Parameter Options Orchestration Action:
  1. Navigate to Admin > Orchestration Actions. 
  2. Scroll to Generated Parameter Options section and click Add an Action.
  3. Select ‘CloudBolt plug-in’ for the action type, and add a new CloudBolt plug-in.

A regenerated options Parameter dependency where you can select one value for the control field and one for the dependent field.

Python
def get_options_list(field, control_value=None, **kwargs):
    """
    A plug-in for regenerating 'Species' options based on the value of the controlling field 'Genus'
    field: This is the dependent field.
    control_value: The value entered on the form for the control field. This value determines what will be displayed in "field"
    """
    options = []

    if control_value == 'Acer':
        options = [
            ('rubrum', 'rubrum'),
            ('saccharinum', 'saccharinum'),
            ('macrophyllum', 'macrophyllum')
        ]
    elif control_value == 'Sequoia':
        options = [
            ('magnifica', 'magnifica'),
            ('langsdorfii', 'langsdorfii'),
            ('giganteum', 'giganteum')
        ]
    return {
        'options': options,
        'override': True
    }

[+] Example 2: Using Multiple Choice Fields
[-] Hide
A regenerated Parameter options dependency where you've set 'allow_multiple' to True on the control field. In this case, you'd need to use the control_values kwarg to accept a list of values from the controlling field.

Python
def get_options_list(field, control_value=None, control_values=[], **kwargs):
   """
   A plug-in for building a list of options for 'Plants'
   based on the values selected for the controlling field 'Plant Type'
   field: This is the dependent field.
   control_value: You won't need to use in this case, but is still required in the kwargs in the function definition.
   control_values: A list of values selected on the form for the controlling field.
   """
   options = []

   if 'Trees' in control_values:
      options.extend([
         ('red_maple', 'Red Maple'),
         ('coastal_redwood', 'Coastal Redwood'),
         ('douglas_fir', 'Douglas Fir'),
         ])

   if 'Shrubs' in control_values:
      options.extend([
         ('red_twig_dogwood', 'Red Twig Dogwood'),
         ('wild_lilac', 'Wild Lilac'),
         ('summersweet', 'Summersweet'),
      ])

   return options
