Minimizing Technical Debt with Orchestration Actions

Learn how to minimize technical debt and automatically remove failed provisioned servers using Orchestration Actions.

Introduction

Technical debt is the trail of digital breadcrumbs left behind as we decommission infrastructure solutions. When a Server has reached the end of its lifecycle, it is decommissioned. But what happens to all of the other dependencies that make sure of the Server functions? The most common debt is the firewall rule. 

CloudBolt provides Orchestration Actions to enable you to address your technical debt.

Prerequisites

  • Access to and understanding of Orchestration Actions.
  • Understanding of the Provisioning Job.

Minimize your Technical Debt with the "Delete Server" Orchestration Hook Point

For every Provision Action, there is a Decommission Action counterpart. While provisioning, CloudBolt will store data from Provision Actions that will later trigger its decom counterpart, removing the technical debt.

Add the "Decom" Orchestration Action

  1. In CloudBolt, navigate to Admin > Orchestration > Actions > Orchestration Actions.
  2. From the Job/Process dropdown, select Delete Server.
  3. From here, view the related "Delete Server" trigger points. The most commonly used is "Post-Delete".
  4. Enable an Action here to allow it to run after the Server record has been deleted from CloudBolt.

Free up a Firewall Rule Automatically

When a Server gets deleted by a User, CloudBolt should automatically free up a firewall rule.

To address this use case, add a Python Plug-in at the Post-Delete hook point to make an API call to SonicWall and delete the Server from the SonicWall.

Automatically Clean up Failed Servers

As a CloudBolt Admin User, I want CloudBolt to automatically clean up the Servers that have failed, so that orphaned Servers are not left in the instance.

In this case, when Provision Jobs fail, CloudBolt automatically deletes the Server from the resource technology. The out-of-the-box Decom script does not touch the firewall and clean up related configurations, adding a Post-Delete Server is recommended.

To address this use case, complete the following:

  1. In CloudBolt, click Admin > Orchestration > Actions > Orchestration Actions.
  2. From the Job/Process dropdown, select Provision Server.
  3. Scroll to Post-Provision.
  4. Toggle "Automatically decom server when provisioning fails" to Enabled.
