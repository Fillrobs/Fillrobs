OS-Specific Requirements

Some operating systems require specific configuration in the template or Environment to work properly.

*nix Servers using SSH

Learn how to run Remote Scripts on *nix (Linux, Mac OS X, etc) Servers using SSH.

Running Remote Scripts using SSH

In order to run remote scripts using SSH:

  • The VM’s IP address must be routable from the CloudBolt server. Often this involves use of a public IP on public cloud environments.
  • CloudBolt must know the IP address of the server, usually because it was provisioned with CloudBolt or the Resource Handler is able to provide it to CloudBolt.
  • TCP port 22 must be open. You can test this by SSH’ing from the CB server to your VM.
  • sshd must be installed and running on the VM.
  • openssh-clients (on CentOS/RHEL), openssh-client (on Debian/Ubuntu), or the equivalent package that provides the scp client must be installed on the VM.

Windows Servers using WinRM

Running scripts on Windows servers on most technologies uses WinRM (Windows Remote Management). CloudBolt can also be configured to run scripts on VMware Windows servers with WinRM rather than VMware tools, as described above.

Prerequisites

  • The VM’s IP address must be routable from the CloudBolt server. Often this involves use of a public IP on public cloud environments.
  • CloudBolt must know the IP address of the server, usually because it was provisioned with CloudBolt or the Resource Handler is able to provide it to CloudBolt.
  • For any server using WinRM, firewalls and permissions within the guest must be configured to allow connections via WinRM.
  1. From the guest Windows OS, launch “Local Group Policy user interface” by typing gpedit.mscin the run prompt.
  2. Navigate to Local Computer Policy > Computer Configuration > Administrative Templates > Windows Components > Windows Remote Management (WinRM) > WinRM Client.
  3. Enable Trust the remote machine. 
  4. In the settings window, add the CloudBolt server name in the “trustedhostedlist” field.
  5. Enable Allow unencrypted traffic under the “WinRM Client” menu.
  6. Enable Allow basic authentication and Allow unencrypted traffic under the “WinRM Service” menu.
  7. Open the Control Panel, go to System and Security > Check Firewall Status > Advanced Settings > Inbound Rules and search for "Windows remote management" (HTTP-in). Double click it, go to advanced and make sure public is enabled/checked.
  8. Open PowerShell in the guest Windows OS and run Configure-SMRemoting. Answer yes to all the prompts to make the VM fully ready for WinRM access.
  9. Add the server to a security group that allows TCP connections on ports 5985 and 5986. In the case of an AWS server, that means going to the Amazon EC2 console and adding the server to an appropriate security group. In the case of Azure, those ports should be open by default in the server’s Network Security Group inbound security rules.

Using WinRM with Encryption

If you want CloudBolt to always use encryption when executing remote scripts with WinRM then you must add the following entry to your /var/opt/cloudbolt/proserv/customer_settings.py: ENCRYPT_WINRM = True. Note that this setting affects all WinRM based remote script executions, including the interactions between CloudBolt and any Hyper-V resource handlers that might have been configured.

At this time, command line arguments are not supported when running scripts on Windows servers.

Using WinRM with CredSSP

CredSSP can be enabled on a server-to-server basis in CloudBolt. CredSSP provides an encrypted Transport Layer Security Protocol channel, and delegates the user’s credentials from the client to the target server for additional remote authentication within a Windows domain. Please note that this is not constrained delegation, CredSSP passes the user’s full credentials to the server without any constraint. To enable CredSSP auth for a given Windows server, add the ‘Supports CredSSP’ Parameter to the Server and set it to True.

For additional info on CredSSP, see the Microsoft documentation.

Using WinRM with Kerberos

Kerberos is the native authentication protocol for Microsoft Windows servers/domains. It provides strong authentication for client/server applications using key cryptography. Kerberos can be used as the transport type with WinRM, providing a trusted authentication to the domain and giving access to all domain resources available to the account.

Basic Concepts

The client will request an authentication ticket (TGT) from the Key Distribution Center (KDC). The KDC will verify the credentials and return an encrypted TGT and session key. Ticket Granting Services (TGS) encrypts the secret key. The client stores the TGT until it expires, at which point another TGT will need to be requested. The default expiration time in the sample is five (5) days.

 DNS Server Configuration

The DNS servers configured on the CloudBolt appliance must be the same as the DNS servers used for the KDC. Otherwise, the domain will never show when a connection is attempted.

Enable Kerberos using the Parameter

The Kerberos Parameter is available out of the box in version 9.4.1 and greater. It can be enabled on the Group, Environment, or Blueprint build item. The most common scenario is at the Environment level. To enable Kerberos authentication for a given Windows server:

  1. Click Admin > Environments and select one.
  2. Click the Parameters tab and click +Add.
  3. Check the box next to Supports Kerberos and click Add.
  4. In the Options column, click the pencil icon to Edit.
  5. Click the dropdown to change False to True and enable the parameter on the Environment.

Configuring Kerberos on CloudBolt 

To configure Kerberos on CloudBolt, you will need to modify the following file, located in this directory: vi/etc/krb5.conf. Enter the example configuration below to get started. Change the example names to the paths you are using.

The FQDNs are case sensitive. Maintain the letter cases as shown in the below example for your directories.
The default ticket lifetime in this example is set to five (5) days. When the ticket lifetime expires, request a new ticket to continue using Kerberos.

Copy Code
Code
# Configuration snippets may be placed in this directory as well
includedir /etc/krb5.conf.d/

[libdefaults]
default_realm = MY.DOMAIN.COM
dns_lookup_realm = false
ticket_lifetime = 5d
renew_lifetime = 7d

[realms]
MY.DOMAIN.COM = {
    kdc = ad-server.my.domain.com
    admin_server = ad-server.my.domain.com
    default_domain = my.domain.com
}

[domain_realm]
.my.domain.com = MY.DOMAIN.COM
my.domain.com = MY.DOMAIN.COM

Ticket Generating Ticket Initialization

CloudBolt does not currently manage regenerating or renewing the TGTs.

The account you choose should have access to all of the necessary resources within that domain. 

  1. Run the following command in a terminal window to generate a TGT. Replace "Administrator" in this example with your domain account name (case sensitive): kinit Administrator.
  2. Enter the password. If successful, you will receive a properly formatted User Principal Name (UPN) for Kerberos authentication. Only this name will be required for Kerberos authentication moving forward.
A non-expired TGT must be associated with the UPN for Kerberos authentication to succeed.
Run this command to view details on all tickets that have been generated: klist.
