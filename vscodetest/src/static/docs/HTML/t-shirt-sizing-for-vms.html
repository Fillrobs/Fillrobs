T-Shirt Sizing for Virtual Machines

Learn how to provide friendly compute sizing options when ordering Servers.


Use Preconfigurations to combine a group of options during the ordering process in order to align with corporate standards, and provide a friendly set of technology-agnostic options to choose from when ordering a Server.

  • T-shirt Sizing Examples with Steps
  • AWS Instance Types
  • VMWare vCenter
  • Azure Node Sizes

T-shirt Sizing Examples with Steps

Refer to the following examples  for AWS and vCenter of how to use t-shirt sizing for VMs in CloudBolt.

AWS Instance Types

A CloudBolt customer has standardized on a set of VM sizes in the form of Small, Medium, and Large across all CloudBolt Environments; however, AWS provides users with a choice of over 50 AWS-specific instance types.

While the company's IT Admin can limit the number of available instance types in all AWS environments to a subset of choices, the goal is to provide standard names across AWS, vCenter, and other public clouds.
A “small” instance in a vCenter-backed environment should be as close as possible to a “small” instance in AWS.

After researching the available instance types, IT determined that their corporate standards, when translated to AWS, are roughly as follows:

Option Value
Small (2CPU 4GB) t3.medium
Medium (2CPU 8GB) t3.large
Large (4CPU 16GB) t3.xlarge

Create the Pre-Configurations

Resource Handlers may use different Parameter names for the VM size. Azure uses node_size and GCP uses gcp_node_size.
  1. Under Admin > Preconfigurations, create a new pre-config called “Server Size” with a name of “aws_server_size”.
    • The name is important as it makes it easier to identify this pre-config when added to an Environment.
  2. Add the instance_type Parameter to the new pre-config.
  3. Add options for Small, Medium, and Large.
  4. For each option, enter the corresponding instance type name.
  5. Set the order values as numbers increasing with size, e.g. 1, 2, 3.
  6. Save the new pre-config.

Configure Environment to use Pre-Configurations

  1. On the AWS Environment, select the Parameters tab, and click the “Add” button.
  2. Find the Parameter labeled “Server Size” with the name “aws_server_size” and add it to the environment.
  3. Test your Environment by ordering an ad-hoc Server. You should see a Server Size form element with Small, Medium, and Large options.

VMWare vCenter

IT wants to provide the Support team with easy access to a vCenter environment. As part of their self-service and standardization initiative, they’ve standardized on t-shirt sizes across all their CloudBolt Environments for VM requests as follows:

Small Medium Large
2x CPUs
4GB RAM
25GB Disk 		2x CPUs
8GB RAM
50GB Disk 		4x CPUs
16GB RAM
100GB Disk

This is the perfect use-case for CloudBolt Pre-Configurations!

Create the Pre-Configurations

  1. Under Admin > Preconfigurations, create a new pre-config called “Server Size” with a name of “vcenter_server_size”.
  2. The name is important as it makes it easier to identify this pre-config when adding it to our Environment.
  3. Add the mem, disk, and cpu_cnt parameters to the new pre-config.
  4. Add options for Small, Medium, and Large.
  5. For each option, enter the corresponding CPU count, Memory (in GiB), and Disk (in GiB).
  6. Set the order values as numbers increasing with size, i.e. 1, 2, 3.
  7. Save the new pre-config.

Configure Environment to use Pre-Configurations

  1. On the vCenter Environment, select the Parameters tab, and click the “Add” button.
  2. Find the Parameter labeled “Server Size” with the name “vcenter_server_size” and add it to the Environment.
  3. Test your Environment by ordering an ad hoc Server. You should see a Server Size form element with Small, Medium, and Large options.

Azure Node Sizes

Similar to the AWS use-case. IT wants to provide sizes for specific Azure Node Sizes. However, on the standard order form, Azure Node Sizes are generated parameters determined by the selected Storage Type. Once IT determines their corporate standards for Azure Sizing, they verify the necessary Storage Type for each chosen Node Size and continue to create the Preconfiguration.

Disable the orchestration actions that are controlling your Generated Parameters

  1. Navigate to Admin > Orchestration Actions and scroll down to the “Generated Parameter Options” hook point under the “Other” category.
  2. Toggle off (Disable) the actions controlling your parameters.
  3. For Azure, these are the actions controlling “node_size” and “storage_type”

Create the Pre-Configurations

  1. Under Admin > Pre-configurations, Create a new pre-config called “Server Size” with a name of “azure_server_size”.
    • The name is important, as it makes it easier to identify this pre-config when adding it to our Environment.
  2. Add the “node_size” parameter and the “storage_type” parameter to the new pre-config.
  3. Add options for Small, Medium, and Large.
  4. For each option, enter the corresponding node size name and storage type.
  5. Set the order values as numbers increasing with size, i.e. 1, 2, 3.
  6. Save the new pre-config

Configure Environment to use Pre-Configurations

  1. On the Azure Environment, select the Parameters tab, and click the “Add” button.
  2. Find the Parameter labeled “Server Size” with the name “azure_server_size” and add it to the Environment.
  3. Test your Environment by ordering an ad hoc Server. You should see a Server Size form element with Small, Medium, and Large options.

Re-enable the Generated Parameter Actions for use in ad-hoc Azure Sizing environments

  1. Navigate to Admin > Orchestration Actions and scroll down to the “Generated Parameter Options” hook point under the “Other” category.
  2. Toggle on (Enable) the actions controlling the parameters.
  3. For Azure, these are the actions controlling “node_size” and “storage_type”.
