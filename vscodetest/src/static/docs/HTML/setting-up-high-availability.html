Setting up High Availability

Before You Begin

Review the Appliance Reference Architecture and follow these one-time steps below for all existing servers:

  1. Copy the Secret Key from the initial CloudBolt appliance to any CloudBolt Web Servers and/or Job Engines that are provisioned later.
    1. The Secret Key is used to encrypt and decrypt sensitive fields on the CloudBolt database.
    2. Log in to the initial CloudBolt appliance using a SCP utility such as WinSCP, FileZilla, Cyberduck, etc.
    3. Copy the Secret Key: 
      Copy Code
      Bash (Unix Shell)
      scp /var/opt/cloudbolt/secrets/secret-key-for-apache.bin
      root@cb-ha2:/var/opt/cloudbolt/secrets/
  2. Copy the UNIQUE_TOKEN section of /var/opt/cloudbolt/proserv/customer_settings.py from the initial CloudBolt appliance to the customer_settings.py file to any CloudBolt Web Servers and/or Job Engines that are provisioned later.

Web Server

CloudBolt uses the Apache HTTP server, which runs as the httpd service on the CloudBolt appliance.

Load Balancer

If you are using multiple Web Servers, we recommend putting them behind a Load Balancer. This ensures that multiple Web Servers share an IP address.

  • Refer to Load Balancer Requirements.

Configuring Web Servers

If this machine is going to be a CloudBolt Web Server, it will notneed to run either a Job Engine nor the Database service.

To create a web server, clone an existing CloudBolt Web Server or run the CloudBolt installer on a new VM.

Each CloudBolt web server needs the source code in /opt/cloudbolt/, as well as access to the Shared Directories described in CloudBolt Appliance Reference Architecture.

Then, perform the following steps:

  1. Disable the Job Engine
  2. Disable PostgreSQL
  3. Update Database Connection to the External Database

Disable Job Engine

Disable the Job Engine on the Web Server.

  1. In the Job Engine supervisord Configuration File, located at/etc/supervisord.d/jobengine.conf, perform the following:
    1. Change the line that begins with autostart to: 
      Copy Code
      Bash (Unix Shell)
      autostart=false
       
    2. Change the line that begins with autorestart to: 
      Copy Code
      Bash (Unix Shell)
      autorestart=false
  2. Use the following command to tell supervisor to reload its configuration files: 
    Copy Code
    Bash (Unix Shell)
    supervisorctl reload

Disable PostgreSQL

postgresql-14 is the daemon that runs the PostgreSQL database and is not necessary on the instance that is connecting to an external database.

Purpose Command
Turn off PostgreSQL
Copy Code
Bash (Unix Shell)
systemctl stop postgresql-14
Prevent PostgreSQL from starting when this CloudBolt appliance reboots
Copy Code
Bash (Unix Shell)
systemctl disable postgresql-14

Update Database Connection to the External Database

If this instance is connecting to an external database, update the database settings to point to the external database:

  1. Go to settings_local.py file: 
    Copy Code
    Bash (Unix Shell)
    #Example using vi to view DATABASES section from settings_local.py
    vi /opt/cloudbolt/settings_local.py
  2. Copy the DATABASES section from the settings_local.py file to be pasted into the customer_settings.py file
  3. Go to customer_settings.py file:
    Copy Code
    Bash (Unix Shell)
    #Example using vi to edit customer_settings.py
    vi /var/opt/cloudbolt/proserv/customer_settings.py
  4. Paste the DATABASES section into the customer_settings.py file
  5. Update the HOST, USER and PASSWORD keys in the DATABASES section to reflect accordingly to your external database server: 
    Copy Code
    Bash (Unix Shell)
    #Example
    HOST: <ip-or-fqdn-of-externalDB>
    USER: <user-externalDB>
    PASSWORD: <password-externalDB>

Job Engine

CloudBolt runs its Job Engine through supervisor, which can be controlled through supervisorctl commands. The Job Engine process(es) can be controlled by editing the configuration file at /etc/supervisord.d/jobengine.conf.

  1. Ensure there are no active Jobs running.
  2. Stop all Job Engine process(es): 
    Copy Code
    Bash (Unix Shell)
    supervisord stop jobengine:*

Configuring Job Engine servers

If this machine is going to be a CloudBolt Job Engine Server, it will notneed to run either the Web service nor the Database service.

To create a Job Engine server, clone an existing CloudBolt Job Engine server or run the CloudBolt installer on a new VM.

Each CloudBolt Job Engine server needs the source code in /opt/cloudbolt/, as well as access to the Shared Directories described in CloudBolt Appliance Reference Architecture.

Then, perform the following steps:

  1. Disable the Web Service
  2. Disable PostgreSQL
  3. Update Database Connection to the External Database

Disable the Web Service

Turn off the CloudBolt Web Server processes by stopping or disabling the httpd using the following commands:

Purpose Command
Stop the Web Server processes
Copy Code
Bash (Unix Shell)
systemctl stop httpd
Prevent Web Server processes from starting when this CloudBolt appliance reboots
Copy Code
Bash (Unix Shell)
systemctl disable httpd

Disable PostgreSQL

postgresql-14 is the daemon that runs the PostgreSQL database and is not necessary on the instance that is connecting to an external database.

Disable Orphaned Job Cleanup

On each Job Engine server, add the following line to the file: /var/opt/cloudbolt/proserv/customer_settings.py

  • Go to the customer_settings.py file:
    Copy Code
    Bash (Unix Shell)
    #Example using vi to edit customer_settings.py
    vi /var/opt/cloudbolt/proserv/customer_settings.py
  • Add the following line to this file:
    Copy Code
    Bash (Unix Shell)
    FEATURE_REGISTRY["jobengine::cancel_orphan_jobs"] = False

Refer to Job Engine documentation for further configuration information on the Job Engine, including Job filtering.

Database

We recommend you use a hosted PostgreSQL service if you are unfamiliar with PostgreSQL administration. However you can setup a PostgreSQL cluster or replication topology as desired to support CloudBolt.

For example, PostgreSQL database servers can be configured to support one-way replication or configured as a redundant cluster. See PostgreSQL Documentation for more information on configuring replication.
