Container Orchestrators

A CloudBolt Container Orchestrator connects CloudBolt to an external Container Orchestrator. CloudBolt can use this to manage containerized objects like Pods and Services via CloudBolt Orders and Actions.

CloudBolt integrates with Kubernetes to manage your clusters. CloudBolt currently supports:

  • Native Kubernetes
  • Google Kubernetes Engine (GKE) 

If you are interested in support for other container orchestrators or have other needs for managing containers, please contact us at support@cloudbolt.io.

Kubernetes Overview

Kubernetes is the industry standard in Container Orchestration. Kubernetes offers many benefits to CloudBolt users:

  • Deployment simplicity: containerized apps and succinct deployment manifests reduce the time and complexity to deploy and update applications.
  • Resource optimization: Kubernetes utilizes the resources available in a given cluster by scheduling as many apps on a given node without sacrificing performance.
  • Orchestration of containers on multiple hosts. This allows for disaster recovery and blue-green testing.
  • Better control of CI/CD pipeline, which allows for controlled standards across multiple build environments. This reduces disparities between dev and production environments.

Discovering a New Container Orchestrator in CloudBolt

  1. Go to the Admin page.
  2. Click External Sytems.
  3. Click Container Orchestrators.
  4. Click Add a container orchestrator.
  5. Choose Kubernetes.
  6. Provide the details CloudBolt needs to connect to Kubernetes.
  7. Click Create.

Now you can add Container Objects to Blueprints in CloudBolt to deploy new Kubernetes apps via CloudBolt Orders.

Deploying a New Container Orchestrator

Kubernetes Blueprint

The Content Library contains a “Kubernetes Cluster” Blueprint to import. This allows a Blueprint Admin to deploy a multi-node Kubernetes cluster. The “Kubernetes Cluster” Blueprint can be deployed to any Resource Handler.

Additional setup tools are required for this Blueprint only for versions 9.1 and older. See Additional Tools.

There are two main use cases for the “Kubernetes Cluster” Blueprint:

  1. Use it on its own to deploy a Kubernetes cluster. This is useful for IT Admins who want to provide multiple standalone clusters into which different apps can be deployed.
  2. Use it as a sub-Blueprint along with Container Objects that have Deploy to new cluster enabled. This will result in a Blueprint that provisions a new Kubernetes Cluster and deploys your app to that Cluster.

Import the “Kubernetes Cluster” Blueprint:

  1. Navigate to the Content Library.
  2. Search for Kubernetes, and select the “Kubernetes Cluster” Blueprint.
  3. Import the Blueprint into your Catalog.

Choose a minimum of 2 CPU and 4 GB for memory. You may choose multiple servers.

The first server created will default to a node with API server, also known as master node. This could take up to 20 minutes depending on the number of deployed servers. Each server after the first will have its own service node.

Additional Tools:

RKE: In 9.2 and greater, RKE is automatically configured in CloudBolt. The “Kubernetes Cluster” Blueprint uses RKE to configure and deploy the Kubernetes cluster. The RKE executable is stored in var/opt/cloudbolt/kubernetes/bin. For more information on RKE, see RKE Docs.

kubectl: The “Kubernetes Cluster” Blueprint uses the kubectl CLI tool to help manage new Kubernetes clusters. kubectl is included in CloudBolt versions 9.2+. For more information on kubectl, see kubectl Docs.

Deploying Apps and Clusters

The “Kubernetes” Blueprint can be used as a Sub-Blueprint to deploy Containerized Objects to dedicated Kubernetes clusters.

  1. Navigate to the Content Library.
  2. Import the “Template: Deploy Containerized App with Kubernetes Cluster” Blueprint.
  3. Choose whether to enable Deploy new cluster (see below).
  4. Follow the steps below for deploying to an existing cluster or a new cluster, depending on the action in step 3.

When Deploy new cluster is disabled on a Container Object, the user will choose an existing cluster to which the Blueprint will deploy. Make sure a Kubernetes cluster is already configured for each Container Object.

  1. From the Catalog, click the “Template: Deploy Containerized App with Kubernetes Cluster” Blueprint, then Manage Blueprint.
  2. Click Build.
  3. Click the pencil icon to open the Edit Blueprint Item window.
  4. Disable Deploy new cluster.
  5. Select one or more existing Deployable Clusters from the drop down menu.
  6. Remove any “Kubernetes Cluster” service items from the Blueprint.
If you don’t see the + Container Object button on the Build tab, it means that Container Objects have no viable destination. Either import the “Kubernetes Cluster” Blueprint from the Content Library, or manually configure a Container Orchestrator from the Admin page.

New Cluster

When Deploy new cluster is enabled, CloudBolt will create a Kubernetes cluster in a new environment, and deploy this containerized object to that cluster.

  1. With the Kubernetes Blueprint selected, click Build.
  2. Click the pencil icon to open the Edit Blueprint Item window.
  3. On the Edit Blueprint Item window, enable Deploy new cluster.
  4. Make modifications to the kubernetes YAML to deploy the app to a new cluster.

Ensure you have imported the “Kubernetes Cluster” Blueprint from the Content Library and added a Kubernetes Cluster as the first build item of the Blueprint.

Configuring Containers

Adding Containers to a Blueprint

Kubernetes objects are deployed from a YAML-formatted specification known as a config file. CloudBolt can automatically generate this config file from parameters you define, or you can provide your own file via upload or a remote URL.

Auto-generated config files are currently limited to the Pod object type and a few parameters. For other object types, please see examples in the Kubernetes documentation for how to create a config file.

Templating with Config Files

When you provide your own config file, you can templatize it using file inputs. CloudBolt will automatically scan the file contents for template style variables like {{ variable }} and will automatically create file inputs for each variable it discovers. Then, an admin can set default values for those file inputs when editing the Blueprint Item where the config file is used. When a user goes to order the Blueprint containing the Kubernetes object, that default value will be used unless they enter a different value on the order form. Before actually using the config file to deploy the Kubernetes object, CloudBolt will substitute the given values into it and use the result as the specification.

In addition to file inputs, you have access to {{ group }}, {{ job }}, {{ blueprint_context }} and other standard CloudBolt context. See Action Context for more details.

As an example of using CloudBolt context, here is how you could assign a Kubernetes namespace to a CloudBolt group:

  1. Under Admin > Parameters, create a parameter with a name of kubernetes_namespace.
  2. Add the parameter to your group and add your Kubernetes namespace as an option.
  3. Under the metadatasection of your YAML file, add namespace: {{ group.kubernetes_namespace }}

Viewing and Editing Containers

If you deploy a resource with Kubernetes objects, a Kubernetes tab will appear on the resource’s detail page listing the objects that were created. If you have the Change Attributes permission for the resource, you can also edit the YAML definition of the deployed object, similar to the kubectl editcommand line tool.

All Kubernetes objects for an entire cluster can viewed and edited from the Deployed Objects tab on the container orchestrator’s detail page. This page is only available to CloudBolt admins. Objects deployed outside of CloudBolt cannot be viewed at this time.

Deleting Containers

When you delete a resource, CloudBolt will also delete any container objects that were associated with that resource. CloudBolt will also make an effort to delete any sub-objects, such as Kubernetes Pods created as part of a Deployment. However, there may be some object types where sub-objects can’t be detected. If you encounter this, please contact us at support@cloudbolt.io.
