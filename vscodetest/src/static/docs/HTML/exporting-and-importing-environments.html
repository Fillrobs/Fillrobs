Exporting and Importing Environments

Environments can be exported as files in order to be easily reused across CloudBolt instances. Exported Environments are typically stored as zip files containing JSON-formatted metadata about the Environment.


Export an Environment from its details page in the UI or using the API.

Exporting Environments in the UI

The Environment will be exported as a .zip file to your local machine. Exports will also contain all the necessary details on any Environment Parameters, Limits, OS Builds, and Networks. The import process will look for existing matching objects to use and create new ones if none are found.

  1. In CloudBolt, click Admin > Integrations > Environments.
  2. Click an Environment name.
  3. Click Export...
  4. (Optional) Provide a password for encrypting and decrypting secrets. See Secrets Stored in Environments.
  5. Choose sanitized or instance-specific format.

Importing Environments in the UI

  1. On the Environments page, click Add an Environment > Upload Environment.
  2. Choose the .zip file created from the Export process.
  3. (Optional) Enter the password created during the Export process.
  4. Choose to Replace Existing Environments or create two Environments with the same name and different Global IDs.
  5. Click Save.

Exporting Environments using the API

Global IDs are installed by default in versions 9.1 and newer. Users of older versions must Transition to Global IDs.
  1. Before you begin, find the Global ID of your Environment. Navigate to Environments > Overview and locate the Global ID field.
  2. On your CloudBolt with the Environment you wish to export, run one of the following command(s). Choose Sanitized or Instance-specific.
Copy Code
Python
# Sanitized format
/api/v2/environments/ENV-globalid/export/

# Instance-specific format
/api/v2/environments/ENV-globalid/export/?instance-specific-info=true

Where ENV-globalid is replaced by the Global ID value of your Environment. Example: ENV-x57nm3w2.

Importing Environments using the API

On your CloudBolt instance that will receive the imported Environment, run the following, and provide an export .zip file.

Copy Code
Python
# Imports the Environment without replacing existing Environment of the same name.
/api/v2/environments/import/

# Replaces an Environment of the same name if one is found.
/api/v2/environments/import/?replace-existing=true

Sanitized and Instance-specific Formats

  • Sanitized format: This version will omit potentially sensitive information or details that may not make sense when importing into a different CloudBolt, such as Groups, the associated Resource Handler, and Global IDs.
  • Instance-specific format: This format is useful for migrating content between your own CloudBolt instances. For example, from "dev" to "prod". This format may include potentially sensitive information such as OS Builds and Groups.

Global IDs During Export/Import

If you’re relying on the Global ID feature for referencing Environments through the API and want to maintain consistency across CloudBolt instances, you’ll generally want to select Replace existing when uploading Environments. If Replace existing is not selected and an Environment already exists with the same name as the export, two Environments will exist with different Global IDs on one CloudBolt.

Secrets Stored in Environments

When an Environment is selected for export, any secrets (Parameter values with the type ‘password’) stored in that Environment can be encrypted with a user-designated password. When the Environment is imported, another prompt will appear to enter the user-designated password. This decrypts secrets when the Environment is imported.

This is useful for maintaining security when sharing exported objects across different CloudBolt instances.

If a user-designated password is not entered in the prompt, a default key will be used. In this case, the exported Environment’s secrets can only be decrypted when imported to the same CloudBolt instance it was exported from.
